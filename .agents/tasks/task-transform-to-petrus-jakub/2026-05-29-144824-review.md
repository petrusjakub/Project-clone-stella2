# Rebranding single-page site from Stella Elsya to Petrus Jakub

A single-file HTML site for a Manulife insurance agent was rebranded from "Stella Elsya" (Senior Manulife Agent, Kalimantan Barat) to "Petrus Jakub" (Partner Bisnis Manulife, Jakarta). All identity markers were replaced: name, title, WhatsApp number, location, experience years, profile image, and brand initial. The COT achievement card and MDRT year badges were removed to match the new agent's credentials. Two new WhatsApp contact surfaces were added: a sticky bottom banner and a floating action button. One product link (MUHC) was redirected from WhatsApp to `haipetrus.netlify.app`.

**Watch for:** Residual Stella Elsya references in metadata and asset URLs that still point to `stellaelsya.com` (confirmed). The MUHC product link now opens a different site (`haipetrus.netlify.app`) rather than a WhatsApp conversation, breaking the UX pattern established by every other product card (confirmed). The favicon SVG still carries `aria-label="Stella Elsya"` (confirmed).

## High-level view

The identity swap is thorough in rendered body text but incomplete in metadata. Several `stellaelsya.com` references remain in canonical URLs, Open Graph URLs, social sharing images, and the terms-and-conditions link. These affect SEO indexing and social preview cards.

The profile image was changed from a massive base64-embedded blob to an external `Pj.jpg` file, dramatically reducing HTML size. The `<picture>` wrapper still references WebP sources in a `public/` directory that likely doesn't exist in this deployment context — vestigial but non-breaking.

The MUHC product card now links to `haipetrus.netlify.app` instead of WhatsApp, which is the only card that doesn't open a direct conversation. The button text also differs from all other cards.

Two new fixed-position contact elements (sticky banner + floating FAB) use a `body { padding-bottom: 52px }` rule that may be insufficient on narrow viewports with font scaling, since the banner's actual rendered height depends on content reflow.

## Incomplete metadata replacement

User-facing text (`<title>`, `og:title`, `twitter:title`, `og:site_name`, `savepage-title`, description, keywords) was updated. However, these fields were left unchanged:

```html
<link rel="canonical" data-savepage-href="https://stellaelsya.com/" href="">
<meta property="og:url" content="https://stellaelsya.com/">
<meta property="og:image" content="https://stellaelsya.com/public/profile-pic.jpg">
<meta name="twitter:image" content="https://stellaelsya.com/public/profile-pic.jpg">
<meta name="savepage-from" content="https://stellaelsya.com/">
```

The `og:image` and `twitter:image` still point to Stella's profile photo on her domain — social media previews will show the wrong person. The canonical URL tells search engines this page is a copy of stellaelsya.com, which may cause indexing issues or attribute SEO value to the wrong site.

The two `<link rel="icon">` elements contain base64-encoded SVGs with `aria-label="Stella Elsya"` baked into the encoded data. Low-severity (favicons aren't screen-read) but an incomplete rebranding artifact.

## MUHC product link deviation

All product cards follow a consistent pattern: a "Tanya produk ini" button linking to `wa.me/6287781896087` with a prefilled WhatsApp message. The MUHC card breaks this:

```html
<a class="button primary full" href="https://haipetrus.netlify.app/" target="_blank" rel="noopener">Tanya lebih lanjut gratis</a>
```

The button text also differs ("Tanya lebih lanjut gratis" vs "Tanya produk ini"). A user clicking this lands on a separate Netlify page rather than opening a WhatsApp conversation — it's the only card that doesn't lead to direct contact.

## Sticky banner padding fragility

The `body { padding-bottom: 52px }` rule is a magic number that must stay in sync with the banner's actual height (12px padding top + 12px padding bottom + line height + flex gap). On devices with larger font scaling, the banner may exceed 52px and overlap footer content. Using a CSS custom property or placing the banner outside the normal flow with a spacer element would be more robust.

The WhatsApp SVG path data is duplicated inline in both the banner and the FAB — extracting to a shared `<symbol>` / `<use>` pattern would reduce payload and maintenance burden, though this is minor for a single-page site.

## Footer terms link

The terms and conditions link still points to `https://stellaelsya.com/terms-and-conditions.html`. On a different deployment domain this will either 404 or display Stella's terms. Either create a local T&C page or update the href.
